# mrp-go — Functional Requirements Document

Status: v1 draft. Owner: Rakshith. This is the spec you design and build against —
treat it the way you'd treat a ticket handed to you at work: authoritative on *what*,
silent on *how*. The "how" is yours to decide and record in `docs/adr/`.

## 1. Purpose

A personal production-management / MRP backend (Go + PostgreSQL, React dashboard)
that reimplements the core planning-to-execution loop of a real commercial seiban-style
manufacturing system, at a scale and depth defensible in a staff-level interview.

## 2. Actors

| Actor | Uses the system to... |
|---|---|
| Planner | create production plans, trigger MRP, review exceptions and the order tree |
| Shop-floor operator | report work results against work orders |
| Buyer | review and action purchase requests generated by MRP |
| System (async worker) | execute MRP jobs in the background and report progress |

## 3. Scope

**In scope (v1, the 1-week build):** master data (items/BOM/routings/plants/warehouses),
production plan creation, MRP explosion (BOM tree → orders), netting against on-hand +
safety stock, lot sizing, backward scheduling against a holiday-aware calendar, async job
execution with live progress, work-result reporting with backflush, an append-only
inventory ledger, a stock projection view, purchase request generation, a REST API, a
minimal dashboard, containerized deployment with CI/CD.

**Explicitly out of scope for v1** (stretch goals, not resume-critical — don't build until
core is done and defensible):
- Resource orders / capacity conflict detection (see [prior conversation] — schema hooks
  may be added early per AD-8, but scheduling/UI is deferred)
- QC / process-control orders
- Lot traceability / genealogy
- Multi-plant transfer orders
- Auth/authorization (single implicit user for v1)
- External ERP/accounting integration

## 4. Current State (baseline, already implemented — Day 1)

Schema (`internal/db/migrations/0001_init.sql`), embedded migration runner, scale seeder,
masters CRUD for items/plans. This baseline was built directly (not as an exercise) to
get a working skeleton fast. Everything from here down is yours to design and build.
You may also revisit and refactor the Day 1 architecture later if you decide it doesn't
fit — that's a legitimate exercise too, not a rule you're locked into.

## 5. Functional Requirements

### FR-1 — Master Data
- FR-1.1 Items have a type (`make`/`buy`), lead time, lot-sizing rule, safety stock.
- FR-1.2 BOM headers/lines carry `qty_per` and `process_seq` (which routing step consumes
  the component). The system must reject a BOM containing a cycle — fail fast, no partial
  writes.
- FR-1.3 Routings have ordered steps, each with a resource, setup time, and hours/unit.
- FR-1.4 Plants have a working-hours window and a holiday calendar.

### FR-2 — Production Planning
- FR-2.1 Creating a plan requires an existing **make**-type item, qty > 0, and a due date.
- FR-2.2 Plans are listable/filterable by status.

### FR-3 — MRP Explosion
- FR-3.1 Given a plan, the system explodes the planned item's BOM tree recursively into a
  hierarchy of production orders — one order per make-item node in the tree.
- FR-3.2 Each production order gets one work order per step of that item's routing, in
  sequence, each linked to its predecessor.
- FR-3.3 Each BOM line under an order's item creates a component requirement attached to
  the work order matching that line's `process_seq` — not to the order as a whole.
- FR-3.4 A cyclic BOM must be detected and rejected before any writes occur.
- FR-3.5 Explosion is atomic per plan: all orders are created, or none are.

### FR-4 — Netting & Lot Sizing
- FR-4.1 For buy-type components: `net = max(0, gross − on_hand − safety_stock)`.
- FR-4.2 `net > 0` generates a purchase request; `net == 0` generates nothing.
- FR-4.3 Lot-for-lot orders exactly the net requirement. Fixed-lot rounds up to the next
  multiple of the item's `fixed_lot_size`.

### FR-5 — Backward Scheduling
- FR-5.1 The plan's due date sets the due date of the root production order's *last* work
  order.
- FR-5.2 Each work order's start = its due date − step duration (`setup_hours +
  hours_per_unit × qty`), walked backward across the plant's calendar, skipping holidays
  and non-working hours.
- FR-5.3 A child production order's due date equals the start date of the parent work
  order step that consumes it.
- FR-5.4 A purchase request's `need_by` equals the start date of the earliest work order
  requiring that component.

### FR-6 — Async Execution & Visibility
- FR-6.1 An MRP run executes as a background job — the triggering HTTP request must not
  block on it.
- FR-6.2 Only one active job per plan at a time. A duplicate trigger while one is running
  is **rejected**, not silently queued.
- FR-6.3 Job status (`queued/running/succeeded/failed`) and progress are queryable and
  update live while running.
- FR-6.4 A failed run leaves no partial orders (see FR-3.5) and records an error message.

### FR-7 — Execution / Work Results
- FR-7.1 An operator reports a result (qty good, qty scrap) against a work order.
- FR-7.2 Reporting backflushes component consumption: each component requirement on that
  work order posts a proportional issue movement.
- FR-7.3 A production order's status rolls up to `done` when all its work orders are done.
- FR-7.4 Work results are append-only — corrections are new adjustment movements, not edits.

### FR-8 — Inventory & Visibility
- FR-8.1 On-hand stock is always `SUM(inventory_movements.qty)` — derived, never a stored
  mutable balance (carry forward from Day 1's design decision).
- FR-8.2 A stock projection view: on-hand + scheduled receipts − scheduled requirements,
  bucketed by date, over a configurable horizon, for a given item.
- FR-8.3 The projection must remain fast at benchmark scale — see NFR-2 for the measured
  target; this is where your naive→optimized story lives.

### FR-9 — Purchasing
- FR-9.1 Purchase requests are listable/filterable by status.
- FR-9.2 A request transitions `open → ordered → received`; `received` posts a receipt
  movement.

### FR-10 — API & Dashboard
- FR-10.1 All of the above is exposed as a REST/JSON API.
- FR-10.2 A minimal dashboard shows: plans list, "run MRP" with live status, order tree
  view, stock projection chart.

### FR-11 — Deployment & Operations
- FR-11.1 The app runs as a container; local dev via docker-compose.
- FR-11.2 `/healthz` (liveness) and `/readyz` (readiness, checks DB); graceful shutdown on
  SIGTERM — drain in-flight requests, don't kill an in-progress MRP transaction mid-write.
- FR-11.3 CI runs lint + test + build on every push.
- FR-11.4 The app deploys to one public cloud target with a reachable URL and uptime
  monitoring.

## 6. Non-Functional Requirements (measured, not assumed)

| ID | Requirement |
|---|---|
| NFR-1 | **Measured at bulk-run scale, not single-plan** — a scheduled run processing many plans together (benchmark scale: `-plans 500`), naive synchronous baseline vs. async + batched-traversal optimized version, both measured and documented in `BENCHMARKS.md`. A single plan's explosion is too fast even unoptimized (~18ms for a 23-header tree, measured 2026-07-20) to honestly substantiate a "minutes → near real-time" claim — bulk scale is what makes that number real, and mirrors why the real UMProcess system has a dedicated bulk MRP execution feature (`PmBulkExecutionMrpJobController`) rather than only single-plan runs. Directly backs `docs/RESUME.md` Bullet B. |
| NFR-2 | Stock projection across 50 items over a 2M-row ledger completes in around a second with proper indexing — document the naive baseline and the optimized number, both measured on your machine |
| NFR-3 | Push-to-deployed pipeline completes in under 5 minutes |
| NFR-4 | `/readyz` reports unready (not 200) whenever the DB is unreachable; no request is accepted once shutdown drain begins |
| NFR-5 | Concurrent MRP runs on *different* plans never block each other; concurrent runs on the *same* plan are prevented outright, not raced |
| NFR-6 | All multi-row writes within one explosion are transactional; FK/CHECK constraints enforce integrity (carried forward from Day 1) |
| NFR-7 | FR-3–FR-5 have table-driven tests covering at minimum: single-level BOM, multi-level BOM, cyclic BOM (must error), fixed-lot rounding, a schedule that spans a holiday |

Do not invent numbers for these — every NFR gets a real measurement in `BENCHMARKS.md`,
taken on your machine, reproducible by command.

## 7. Business Rules Quick Reference

- **Pegged, not netted across plans.** This is seiban-style MRP: each plan's explosion is
  independent (FR-3.5's atomicity is per-plan). There is no cross-plan netting of shared
  component demand in v1 — that's a real simplification vs. textbook MRP II, and you should
  be able to say so plainly in an interview, not hide it. Note this is a bigger simplification
  than it first looks: the real source system supports make-to-stock *and* seiban **per
  item** (an `Item__c.MakeToStockFlg__c`-style toggle), not as a system-wide mode — v1 here
  assumes every item is pegged/seiban uniformly, because supporting both would require
  genuinely different netting logic (cross-plan netting for make-to-stock items) that isn't
  necessary to demonstrate the resume-bullet skills this project targets. Name this precisely
  if asked, don't imply the real system only ever worked this one way.
- **Low-level coding isn't needed here** *because* of the pegged model above — components
  aren't processed once across all demand, they're processed once per plan's tree. If you
  ever add cross-plan netting as a stretch goal, low-level coding (FR-3 in the earlier MRP
  explanation) becomes necessary and this rule changes.
- **A component's due date comes from the step that consumes it**, not from the order as a
  whole (FR-3.3, FR-5.3) — this is what makes `process_seq` load-bearing rather than
  decorative.
- **The ledger is the only source of truth for stock** (FR-8.1) — no code path anywhere
  may write a balance column instead of a movement row.
- **BOM is a header/lines split** (`bom_headers` + `bom_lines`), matching the real source
  system's `CompositionPattern__c`/`Composition2__c` structure — a header represents one
  named recipe for an item, letting multiple recipes and recipe-level metadata (name,
  associated routing, active flag) exist without denormalizing across every line. What's
  deliberately *not* carried over from the real system: multiple named patterns per item
  (this schema's seeder only ever creates one `'STD'` pattern), `ChildCompositionPattern__c`
  (the real system lets a BOM line pin a *specific variant recipe* of the child item, not
  just the item itself), `ByProductFlag__c`/`CoProductFlag__c` (co-product/by-product
  tracking), and `EnableStartDate__c`/`EnableEndDate__c` (per-line effectivity dates). All
  real, all deliberately out of scope — name them precisely if asked, don't imply the
  simpler version is the only way UMProcess itself works.

## 8. Open Architecture Decisions — yours to make

Write one ADR (`docs/adr/000N-title.md`, template provided) per decision *before* or *while*
you build the relevant FR. An ADR is short: context, options considered, decision, why.

- **AD-1** — BOM tree traversal: recursive SQL CTE, in-Go recursion after a flat load, or
  both (compare)? Which does FR-3.1 get built on first?
- **AD-2** — Does netting (FR-4) run in the same pass/transaction as explosion (FR-3), or
  as a separate step?
- **AD-3** — Worker pool shape for FR-6: fixed goroutine pool consuming a job channel, or
  bounded concurrency (`errgroup` + semaphore) applied *within* one explosion across
  independent BOM branches — or both, at different levels?
- **AD-4** — How do you guarantee FR-6.2 ("one job per plan")? Postgres advisory lock, a
  partial unique index on `mrp_jobs(plan_id) WHERE status='running'`, or an app-level lock
  map? What breaks if you ever run more than one API instance?
- **AD-5** — Progress transport for FR-6.3: SSE, polling, or WebSocket? Why?
- **AD-6** — FR-8.3's optimization: covering index, a snapshot/materialized table kept in
  sync on write, or both? What's the staleness trade-off of each?
- **AD-7** — Is backflush (FR-7.2) synchronous inside the work-result request, or deferred
  to an async step? What does each choice cost or buy you?
- **AD-8** — If you add resource orders later (deferred per §3), what schema hooks, if any,
  are cheap to add now vs. expensive to retrofit?

## 10. Acceptance Scenarios (seed your table-driven tests from these)

```
Given item "wheel" (make) with BOM 36×"spoke" (buy, on-hand 50, lead time 3d)
  and routing step "assemble" (process_seq 10)
When a plan for 100×"wheel" due day 10 is exploded
Then one production order for "wheel" exists, due day 10
  and its work order "assemble" starts day 8 (10 − 2d lead time)
  and a component requirement for 3600×"spoke" is attached to that work order
  and netting yields 3600 − 50 = 3550 net, triggering a purchase request
  and that request's need_by = day 8

Given a BOM where item A contains B and B contains A
When explosion runs
Then it fails before writing any production_orders row, with a cycle-detected error

Given a plant holiday on day 9
When backward scheduling would land a work order start on day 9
Then the start date shifts to the nearest earlier working day
```
